# 基于知识图谱的中医药两级智能诊疗Agent项目-前端需求文档（优化版）

## 1. 文档定位

- 来源文档：`基于知识图谱的中医药两级智能诊疗Agent项目——新版.docx`
- 文档目标：将原始完整需求整理为前端可直接执行的 `PRD/联调文档`
- 适用对象：
  - 前端开发
  - UI/交互设计
  - 后端联调
  - 测试验收
- 端侧范围：
  - 患者端：`Uni-app` 微信小程序，仅手机端使用
  - 医生端：独立 `Vue3` PC 网页后台，仅浏览器访问

## 2. 项目概览

本项目是一个中医院“线上预问诊 + 线下接诊”一体化诊疗系统，整体由患者小程序、医生 PC 网页后台、后端服务、MySQL 业务数据库、Neo4j 中医药知识图谱、LangGraph 两级分层 AI 诊疗 Agent 组成。

核心业务闭环：

`患者注册建档 -> AI预问诊 -> 推荐科室/挂号 -> 医生接诊 -> 医患实时图文对话 -> AI辅助完善四诊信息 -> 输出辨证和方剂建议 -> 推荐相似药材/医案 -> 医生手动知识库检索 -> 病历/聊天/处方归档`

## 3. 前端范围与边界

### 3.1 本次前端建设范围

- 患者端：
  - 注册建档
  - AI 预问诊
  - 导诊挂号
  - 医患聊天
  - 知识库科普
  - 个人中心与病历查看
- 医生端：
  - 登录与接诊队列
  - 医患对话工作台
  - AI 辅助辨证与推荐弹窗
  - 病历检索管理
  - 个人医案库
  - 数据统计后台

### 3.2 技术边界

- 患者端使用 `Uni-app` 小程序方案。
- 医生端使用独立 `Vue3` PC 后台方案。
- 两端共用同一套后端 API、MySQL 数据、Neo4j 图谱能力和 AI Agent。
- 前端不得直连 Neo4j 或 MySQL，所有能力通过后端 HTTP 接口访问。
- 医生端仅考虑 PC 大屏使用场景，不提供医生小程序端。

### 3.3 数据与权限边界

- 患者端必须实名注册并完成建档后，才能进入完整诊疗流程。
- 患者可修改基础健康档案。
- 患者不可编辑诊疗病历和医生处方，仅可查看。
- 医患聊天、处方记录、门诊病历均需自动归档。

## 4. 建议优先级

以下优先级为基于原文“核心新增功能”整理出的前端落地建议：

| 优先级 | 模块 | 说明 |
| --- | --- | --- |
| `P0` | 患者注册建档 | 诊疗流程入口，生成 `patient_id` |
| `P0` | AI 预问诊 | 患者端核心交互，承接后续挂号 |
| `P0` | 在线挂号 | 形成接诊队列，串联患者端与医生端 |
| `P0` | 医生接诊队列 | 医生侧主入口 |
| `P0` | 医患对话工作台 | 医生端核心页面，集成聊天、病历、AI 输出 |
| `P0` | AI 分步辅助诊疗 | 医生侧核心新增能力 |
| `P0` | 推荐按钮与详情弹窗 | 结构化推荐数据的主要前端承载方式 |
| `P1` | 电子病历检索管理 | 历史数据查询与病历查看 |
| `P1` | 手动知识库检索 | 医生工作台增强能力 |
| `P1` | 个人中心 | 挂号记录、病历、聊天查看 |
| `P2` | 医案库管理 | 上传、对比、相似医案能力 |
| `P2` | 数据统计后台 | 管理与统计视图 |

说明：

- `P0`：必须完成，直接决定主流程是否可用。
- `P1`：应在主流程稳定后完成。
- `P2`：增强与运营类能力，可放在后续迭代。

## 5. 患者端需求

### 5.1 页面清单

| 页面 | 页面目标 | 核心功能 |
| --- | --- | --- |
| 登录注册建档页 | 完成实名建档 | 微信授权登录、基础信息录入、健康档案录入 |
| AI 预问诊对话页 | 完成症状采集与导诊 | 多轮问诊、追加提问、推荐科室、免责提示 |
| 挂号科室/排班页 | 完成挂号下单 | 科室列表、医生简介、时段选择、余号展示 |
| 挂号订单页 | 管理挂号结果 | 查看订单、取消未就诊订单、查看历史挂号 |
| 医患聊天页 | 与接诊医生持续沟通 | 发送文字、上传舌苔照片、查看处方与复诊建议 |
| 个人中心 | 统一查看个人资料和诊疗记录 | 基础档案、问诊记录、挂号记录、病历、处方 |

### 5.2 登录注册与电子病历建档

- 微信授权登录后，强制建档。
- 系统生成全局唯一 `patient_id`，保证一人一档。
- 建档字段包括：
  - 基础身份：姓名、性别、年龄、手机号、身份证
  - 健康档案：药物/中药材过敏史、既往慢病、家族病史、女性经期孕产信息
- 前端需要支持患者修改基础健康档案。
- 前端需要明确区分“可编辑基础档案”与“只读诊疗病历/处方”。

### 5.3 AI 预问诊

- 支持多轮采集：
  - 主诉
  - 症状
  - 舌象
  - 脉象
- 聊天底部保留“追加提问”入口。
- “追加提问”对应 `mode=follow-up`，用于在已有信息基础上继续补充，不重置当前问诊上下文。
- Agent 自动匹配 Neo4j 科室实体，返回：
  - 推荐就诊科室
  - 居家简易调理建议
- 页面全程展示免责说明：线上预问诊仅供参考，不可替代线下医生面诊。

### 5.4 在线挂号

- 预问诊结束后，支持一键跳转挂号页。
- 同时支持用户主动进入挂号流程。
- 挂号页需展示：
  - 科室列表
  - 医生简介
  - 就诊时段
  - 当日剩余号源
- 提交后生成挂号订单。
- 用户可在个人中心：
  - 查看未就诊订单
  - 取消未就诊订单
  - 查看历史挂号记录

### 5.5 医患实时图文对话

- 挂号成功后，开启与接诊医生的专属聊天会话。
- 会话必须绑定 `session_id + patient_id`。
- 患者可发送：
  - 文字描述
  - 身体变化说明
  - 服药感受
  - 舌苔照片
- 系统自动归档全部聊天内容，并同步写入患者电子病历。
- 聊天中需可持续查看医生回复的：
  - 处方内容
  - 药材用法用量
  - 复诊建议

### 5.6 知识库科普

- 支持中药材模糊检索。
- 可查看：
  - 性味归经
  - 禁忌
  - 功效
- 经典医案需做适合普通用户阅读的简化科普展示。
- 药材、方剂可点击跳转到对应科普页面。

### 5.7 个人中心

- 支持修改基础健康档案。
- 展示历史 AI 预问诊记录。
- 展示挂号订单。
- 展示医患聊天记录。
- 展示医生处方。
- 展示完整门诊电子病历。

## 6. 医生端需求

### 6.1 页面清单

| 页面 | 页面目标 | 核心功能 |
| --- | --- | --- |
| 登录页 | 医生账号登录 | 账号密码登录 |
| 接诊队列首页 | 查看待接诊患者 | 列表、筛选、进入接诊 |
| 医患对话工作台 | 完成接诊与 AI 辅助诊疗 | 聊天、病历、四诊录入、AI 输出、推荐弹窗、手动检索 |
| 病历检索管理页 | 查看历史门诊病历 | 多条件检索、单次病历详情 |
| 个人医案库 | 管理医生自建医案 | 上传、对比、相似医案 |
| 系统数据统计后台 | 查看运营统计 | 问诊量、证型、方剂图表 |

### 6.2 登录与角色

- 仅支持医生账号密码登录。
- 角色分级：
  - 管理员
  - 普通医师
- 医生端页面需适配 PC 大屏场景，重点保证：
  - 病历表格可读
  - 侧边栏信息密度可控
  - 弹窗详情可承载较多内容

### 6.3 接诊队列首页

- 首页展示当日所有已挂号患者列表。
- 列表字段包括：
  - 姓名
  - 年龄
  - 挂号科室
  - 预问诊简要症状
  - 就诊状态：待接诊 / 问诊中 / 已结束
- 支持筛选：
  - 科室
  - 初诊/复诊
  - 挂号时间
- 点击“接诊”后：
  - 跳转到医患对话工作台
  - 自动加载患者完整电子病历

### 6.4 医患对话工作台

#### 6.4.1 页面布局

- 右下区域：医患图文聊天框
  - 医生输入框
  - 患者消息展示区
- 右上区域：固定病历侧边栏
  - 基础信息
  - 过敏史
  - 历史辨证
  - 历史处方
- 中间或左中区域：AI 辅助诊疗输出区
  - 辨证结论
  - 治法
  - 推荐方剂
  - 关联药材/医案推荐
- 左下区域：手动知识库检索输入框

#### 6.4.2 基础接诊功能

- 医生可发送文字回复。
- 医生可录入：
  - 方剂
  - 药材用法用量
  - 复诊注意事项
- 医生可手动录入四诊信息：
  - 舌象
  - 脉象
  - 主诉
  - 现病史
- 手动录入的四诊信息需同步传入 Agent。
- 全部对话需自动归档到患者病历。
- 医生可修正 AI 的辨证结论，修正结果保存到个人医案库。

#### 6.4.3 AI 分步辅助诊疗

- Agent 读取当前患者症状与四诊数据后，自动判断信息缺口。
- 需逐条返回下一步采集建议，例如：
  - 缺少脉象信息
  - 未确认是否自汗
  - 影响风寒/风热区分的关键字段缺失
- 信息完整后，Agent 输出：
  - 完整辨证结论
  - 对应治法
  - 推荐方剂
  - 每条结论对应的图谱来源实体或依据

#### 6.4.4 推荐项与弹窗

- AI 输出辨证/方剂后，页面下方自动展示两类推荐项：
  - 相似药材列表
  - 同证型经典医案列表
- 推荐项需渲染为可点击按钮或卡片。
- 点击任一推荐项后，弹出独立详情弹窗。
- 弹窗内容来自 Agent 基于图谱的二次检索与汇总。
- 推荐项展示与详情弹窗应与当前主问诊流程解耦。

#### 6.4.5 手动自定义知识库检索

- 医生可输入自然语言查询指令，例如：
  - “麻黄的功效、配伍禁忌、相似药材”
  - “风寒夹湿证经典医案”
  - “桂枝汤加减方案”
- 输入后触发 Agent 独立检索 Neo4j。
- 此请求不得干扰当前主问诊流程。
- 返回内容需为结构化总结结果，适合前端直接渲染。

### 6.5 电子病历管理

- 支持按以下条件检索患者历史病历：
  - 姓名
  - `patient_id`
  - 证型
  - 就诊时间
- 支持查看单次门诊完整病历。
- 病历展示需表格化呈现：
  - 四诊信息
  - 辨证结果
  - 方剂
  - 医患聊天摘要

### 6.6 医案库管理

- 支持医生上传自建临床医案。
- 上传后同步写入 Neo4j，并关联对应证型实体。
- 支持多医案对比，Agent 自动总结：
  - 用药差异
  - 辨证差异
- 支持相似医案智能匹配。

### 6.7 系统管理与统计

- 支持 Neo4j 图谱基础数据维护：
  - 新增药材实体
  - 修改药材实体
  - 新增证型实体
  - 修改证型实体
- 支持统计图表展示：
  - 问诊量
  - 高频证型
  - 常用方剂

## 7. 核心流程

### 7.1 患者端主流程

1. 微信登录并完成建档
2. 进入 AI 预问诊采集症状
3. 获取推荐科室或主动选择挂号
4. 完成挂号下单
5. 与接诊医生进入实时图文对话
6. 查看医生处方、用药说明和复诊建议
7. 在个人中心查看完整病历和历史记录

### 7.2 医生端主流程

1. 医生登录系统
2. 在接诊队列查看今日挂号患者
3. 点击“接诊”进入医患对话工作台
4. 查看患者既往病历和当前预问诊信息
5. 手动补录四诊信息并触发 AI 辅助诊疗
6. 获取辨证结果、方剂建议和推荐项
7. 点击推荐项查看弹窗详情，必要时发起手动知识库检索
8. 确认方剂并保存至患者电子病历

## 8. 接口与 Agent 对接约束

### 8.1 Agent 调用参数

统一对外函数：

```python
def tcm_agent_chat(
    session_id: str,
    patient_id: str,
    user_input: str,
    mode: str = "normal",
    scene: str = "guide"
) -> str
```

参数定义：

| 参数 | 含义 | 前端使用场景 |
| --- | --- | --- |
| `session_id` | 单次对话会话标识 | 医患聊天、AI 问诊、医生工作台 |
| `patient_id` | 患者全量档案标识 | 建档后所有业务主链路 |
| `user_input` | 用户输入内容 | 聊天、追加提问、手动检索 |
| `mode=normal` | 正常问诊流程 | 首次问诊、常规对话 |
| `mode=follow-up` | 追加提问模式 | 预问诊补充提问 |
| `scene="guide"` | 患者导诊场景 | 小程序 AI 预问诊 |
| `scene="doctor"` | 医生辅助诊疗场景 | 医生工作台 |

### 8.2 医生端结构化推荐返回

当 `scene=doctor` 且辨证完成后，除文本答复外，还会返回结构化推荐数据：

```json
{
  "similar_herb": ["麻黄", "桂枝", "紫苏"],
  "similar_case": ["风寒感冒麻黄汤医案1", "风寒感冒轻症医案2"]
}
```

前端处理要求：

- `similar_herb` 渲染为药材推荐按钮/卡片。
- `similar_case` 渲染为医案推荐按钮/卡片。
- 点击条目后，前端需单独发起请求获取弹窗详情。
- 详情请求不能覆盖当前主聊天记录和主问诊状态。

### 8.3 后端对接规范

- Agent 不直连 Neo4j、MySQL，统一走后端 HTTP 接口。
- 后端需接收前端传入：
  - `session_id`
  - `patient_id`
  - `user_input`
  - `mode`
  - `scene`
- 后端需提供统一知识图谱查询接口，覆盖：
  - 药材
  - 证型
  - 医案
  - 科室
- 后端需支持结构化弹窗详情请求，并透传至 Agent 后返回结果。

## 9. 页面状态与异常处理

### 9.1 必备状态

- 空状态：
  - 无挂号记录
  - 无历史病历
  - 无推荐项
- 加载状态：
  - AI 生成中
  - 推荐项加载中
  - 弹窗详情加载中
- 成功状态：
  - 建档成功
  - 挂号成功
  - 处方保存成功
- 失败状态：
  - 建档失败
  - 挂号失败
  - 聊天发送失败
  - 推荐详情获取失败

### 9.2 异常提示

原文明确要求异常场景需返回友好提示，前端至少需覆盖：

- 模型超时
- 图谱查询无结果
- 参数缺失
- 接口请求失败

建议前端统一错误表达方式，避免将后端原始报错直接暴露给用户。

## 10. 验收清单

### 10.1 患者端验收

- 微信登录后必须先建档，再进入核心业务流程。
- 建档数据能正确保存并再次回显。
- AI 预问诊支持连续对话和“追加提问”。
- 挂号页可完成选科室、选医生、选时段、提交订单。
- 订单页可查看未就诊订单、历史订单，并取消未就诊订单。
- 医患聊天可发送文字并上传舌苔照片。
- 患者可查看医生处方、用法用量和复诊建议。
- 个人中心可查看问诊、挂号、聊天、病历、处方。

### 10.2 医生端验收

- 医生账号可登录并进入接诊队列。
- 接诊队列可按科室、初复诊、挂号时间筛选。
- 点击接诊可跳转工作台并加载对应患者病历。
- 工作台能同时展示聊天区、病历区、AI 输出区、手动检索区。
- 手动录入四诊信息后可同步触发 Agent。
- AI 能返回分步引导、辨证结论、治法、方剂与推荐项。
- `similar_herb`、`similar_case` 可正确渲染并打开详情弹窗。
- 手动知识库检索不会中断当前问诊流程。
- 处方保存后能归档到患者病历。

### 10.3 联调验收

- `session_id + patient_id` 绑定关系正确。
- `scene=guide` 与 `scene=doctor` 能正确区分业务场景。
- `mode=normal` 与 `mode=follow-up` 能正确区分问诊模式。
- 结构化推荐数据字段稳定，前端可按固定键名渲染。
