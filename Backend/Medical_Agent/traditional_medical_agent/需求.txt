基于知识图谱的中医药两级智能诊疗Agent项目——新版完整功能需求说明书

1.适配实训6人分工架构，在原版基础上新增**线上挂号、AI预问诊、标准化电子病历、医患实时对话、医生端AI智能弹窗检索、病历表格展示、手动自定义知识库查询**全流程业务；
2.分层清晰：项目概述→产品载体区分→患者端完整需求→医生PC网页端完整需求→两级Agent改造需求→知识图谱需求→后端需求→前端需求→运维/文档分工对照表；

一、项目整体概述
本项目为中医院线上预问诊+线下接诊一体化系统，整体由**Uni-app微信小程序（患者端）+Vue3PC医生网页后台+FastAPI/Django后端+MySQL业务数据库+Neo4j中医药知识图谱+LangGraph两级分层AI诊疗Agent**构成。
核心业务闭环
患者小程序注册建档→AI预问诊采集症状→线上预约科室挂号→线下医生网页接诊→医患实时图文对话→AI多轮引导完善四诊信息→依托知识图谱输出可溯源辨证+方剂建议→自动推送相似药材/经典医案弹窗→医生可手动自定义查询药材、证型、医案→完整病历、聊天、处方记录归档存档。
两级Agent为医患两端统一底层引擎：
1.患者端：完成AI预问诊、导诊分诊、基础药材科普；
2.医生PC端：临床辅助问诊、逐步引导采集缺失症状、图谱溯源辨证、自动推送关联药材/医案、支持医生自定义知识库检索；
3.所有辨证、药材、医案查询均对接后端封装的Neo4j接口，Agent不直连数据库，分层规范。
技术约束：
-Agent架构：LangGraph两级调度（Supervisor顶层调度Agent+问诊抽取/诊断/自定义查询3个子Agent），诊断Agent一次调用知识图谱获取全部结果（证型、方剂、药材、医案、科室）；
-大模型来源：阿里云DashScope API（通义千问系列）；Supervisor/问诊抽取/诊断使用qwen-max模型，自定义查询使用qwen-turbo轻量化模型；
-前端：患者Uni-app小程序、医生独立Vue3网页后台；
-数据隔离：单院内病历独立存储。

二、产品载体与角色权限划分
2.1患者端载体：Uni-app编译微信小程序（仅手机端使用）
面向普通就诊群众，可进行AI问诊，挂号、保存病历、医患对话必须实名注册建档。
2.2医生端载体：独立Vue3PC网页后台（电脑浏览器访问）
仅医师账号密码登录，具备挂号接诊、医患对话、AI辅助辨证、电子病历管理、知识库检索、自建医案权限；无小程序医生端，电脑大屏适配病历表格、图谱可视化、弹窗详情。
2.3底层共用资源
同一套两级AI智能体、同一套Neo4j中医药知识图谱、同一MySQL业务数据库、统一后端API服务。

4.2.3AI分步辅助诊疗（核心新增功能，两级Agent支撑）
1.多轮信息缺失引导
Agent读取当前医生录入的所有患者症状、四诊数据，自动判断信息缺口，逐条给出下一步采集建议（如“缺少脉象信息，请补充；未确认有无自汗，影响风寒/风热区分”），逐步缩小辨证范围；
2.可溯源辨证与方剂输出
信息收集完整后，Agent调用Neo4j知识图谱，输出完整辨证结论、对应治法、推荐方剂，每条结论标注图谱来源实体（证型、药材、配伍关系），有理有据；
3.自动关联推荐弹窗功能
AI输出辨证/方剂后，自动在页面下方展示两类可点击推荐条目：
-相似药材列表；
-同证型经典医案列表；
点击任意条目，弹出独立详情弹窗，Agent自动检索图谱并总结该药材/医案完整信息展示在弹窗内；
4.手动自定义知识库查询（不受自动推荐限制）
医生在左侧ai聊天框底部输入框自由输入查询指令，例如：
-“麻黄的功效、配伍禁忌、相似药材”
-“风寒夹湿证经典医案”
-“桂枝汤加减方案”
输入后触发Agent单独检索Neo4j，直接返回结构化总结内容，不干扰当前问诊流程。

4.2.4处方记录归档
医生确认辨证方剂后，填写药材剂量、煎服法、服用周期，点击保存自动写入患者本次门诊电子病历，永久留存。

2.多医案对比，Agent自动总结用药、辨证差异；
3.相似医案智能匹配。

3.问诊量、高频证型、常用方剂数据可视化统计图表。

五、两级分层AI诊疗Agent（算法工程师负责，新版改造需求）
5.1整体架构
LangGraph实现两级调度：顶层Supervisor调度Agent + 问诊抽取Agent + 诊断+回复合并Agent + 自定义查询Agent + 医案检索Agent，共4个有效节点。通过Supervisor单次LLM调用判断意图后路由到对应节点。诊断与回复生成合并为单一节点（单次LLM调用内完成"症状标准化→知识图谱查询→科室智能判断→自然语言回复生成"全流程），最大化减少LLM调用次数，提升响应速度、降低成本。知识图谱查询通过封装服务调用neo4j_2数据库（提供更完整的实体关联信息，包括方剂-药材详细组成、证型-方剂关联、过敏冲突方案等）。对接阿里云DashScope API（qwen-max模型）。

5.2顶层Supervisor调度Agent核心能力
1.三参数区分业务场景（原mode基础新增scene场景标识）
-`scene="guide"`：患者小程序AI预问诊导诊场景；
-`scene="doctor"`：医生PC端临床辅助诊疗场景；
-`mode=normal`完整问诊流程/`mode=follow-up`锁定现有信息追加提问；
对外统一函数：
```python
def tcm_agent_chat(session_id: str, patient_id: str, user_input: str, mode: str = "normal", scene: str = "guide", patient_profile: Optional[Dict[str, Any]] = None) -> Dict[str, Any]
```
新增`patient_id`和`patient_profile`参数，用于拉取患者过敏史、历史病历数据参与辨证校验。
2.Supervisor核心能力（单次LLM调用完成意图判断）
-输入：当前用户输入 + 对话历史 + 当前State（symptoms_info等）；
-输出意图分类：`diagnosis`（症状完整进入诊断）/`ask`（症状不完整追问）/`custom_query`（医生手动查询中药/方剂）/`medical_case`（医案检索）/`done`（已完成）；
-【追问原则】只追问缺失的关键信息（如缺舌象就问舌象、缺脉象就问脉象），不要重复已知信息，避免无限循环追问；
3.患者过敏史逻辑
-接收后端传入`patient_profile.allergy_history.herbs`，诊断Agent自动规避过敏药材；
-知识图谱层预过滤过敏方剂，回复层显式提示"患者对XX过敏"信息；
4.会话隔离：`session_id`区分单次对话，`patient_id`绑定患者全量档案；
5.异常兜底：捕获大模型超时、图谱查询无结果、参数缺失、JSON解析错误等异常，返回友好提示文本。

5.3子Agent分工
子Agent1：问诊抽取Agent
1.单次LLM调用完成：症状提取 + 标准化归一（口语→标准中医术语，例如"发寒"→"恶寒"、"没汗"→"无汗"、"腰酸"→"腰痛"等）+ 舌象脉象识别；
2.标准化规则维护在LLM提示词中（SYMPTOM_NORMALIZATION_PROMPT），包括常用口语-标准术语映射表；
3.提取主诉（chief_complaint）、判断is_complete（症状+舌象+脉象都齐则为True）、缺失信息描述；
4.`follow-up`追加提问模式：若state中已有symptoms_info，则跳过抽取直接进入诊断；
5.输出结构化SymptomsInfo：symptoms(标准化后)、tongue、pulse、chief_complaint、is_complete、missing_info。

子Agent2：诊断+回复合并Agent（核心节点）
1.单次LLM调用完成：调用知识图谱 + 智能判断科室 + 生成自然语言回复；
2.知识图谱查询：
-调用`query_kg_for_symptoms(symptoms, allergy_herbs)`获取完整结果；
-返回字段：证型、方剂（含可用/被过敏过滤的两组）、药材列表、医案、科室、警告信息；
3.科室智能判断：
-优先从知识图谱dept实体获取（neo4j_2含此实体时）；
-图谱无科室信息时，由大模型基于症状+证型自动判断（参考常见中医科室：呼吸内科、消化内科、心血管内科、神经内科、妇科、儿科、骨伤科、皮肤科、针灸推拿科、治未病科等）；
4.方剂处理：
-根据证型查询图谱方剂，分数阈值score>=70过滤低相关项；
-解析`treatmentPlans`（available状态）和`notRecommendedTreatmentPlans`（含reason），后者用于解释"图谱检索到但为什么不推荐"；
-读取患者过敏史，自动过滤、高亮过敏药材；
-若所有可用方剂均含过敏药材，则返回"无最佳匹配方剂"；
5.医案检索：Neo4j实体关联检索同证经典医案，自动返回Top3相似医案列表，前端点击弹出详情（精简原始医案，提炼辨证、用药核心逻辑）；
6.回复生成（最重要！）：
-调用大模型（结构化输出`{department, response}`），整合知识图谱结果生成自然语言回复；
-患者场景（guide）：温和亲切语气，像朋友间交流，避免报告式Markdown排版，自动给出就诊科室建议；
-医生场景（doctor）：专业但温和语气，像有经验的同事当面讨论病例；
-【关键约束】当方剂为"无最佳匹配方剂"或"未找到方剂"时：
   a. 明确告诉医生"知识图谱未找到合适的方剂"；
   b. 列出"知识图谱中可考虑的治疗方案（被患者过敏排除）"作为参考；
   c. 建议医生"基于自身临床经验谨慎选择其他方剂"；
   d. 绝对不要自行编造或推荐"麻黄汤"、"桂枝汤"等未在知识图谱中出现的方剂；
   e. 安全第一：患者过敏的药材绝对不能出现在推荐方剂中。

子Agent3：自定义查询Agent
1.接收医生手动知识库检索指令，单独执行图谱检索（`custom_query_kg`）；
2.支持自由输入查询指令，例如："麻黄的功效、配伍禁忌、相似药材"、"风寒夹湿证经典医案"、"桂枝汤加减方案"；
3.返回结构化总结内容，不干扰当前问诊流程。

子Agent4：医案检索Agent
1.由Supervisor识别`medical_case`意图后路由；
2.数据源：独立neo4j医案知识图谱（与neo4j_2分开，专门存储医案-医生-患者-处方关联）；
3.支持按方剂、症状、医生等维度检索相关医案。

5.4Agent输出结构化规范（对接前端弹窗）
当`scene=doctor`诊断完成后，除文本答复外同步返回结构化JSON推荐数据：
```json
{
  "diagnosis_result": {
    "syndrome": "证型名称",
    "prescription": "方剂名称",
    "ingredients": ["药材1", "药材2", ...],
    "department": "推荐科室",
    "allergy_warnings": ["过敏药材1", "过敏药材2"]
  },
  "status": "diagnosed",
  "response": "自然语言回复"
}
```
医生场景额外返回`similar_herb`（相似药材推荐）、`similar_case`（相似医案推荐），前端解析后渲染可点击按钮，点击单独发起Agent查询生成弹窗内容。

六、Neo4j中医药知识图谱（图谱工程师需求）
6.1原有5类实体保留：Symptom症状、Zheng证型、Herb药材、Prescription方剂、MedicalCase医案
6.2关联关系
`Symptom-[:BELONG_TO_DEPT]->Dept`（症状归属就诊科室）
原有关系全部保留：HAS_SYMPTOM、RECOMMEND_PRESCRIPTION、CONSIST_OF、CONFLICT、SYNERGY、HAS_CASE、USED_IN。

七、后端FastAPI+Django（后端工程师完整需求）
7.1MySQL新增数据表
1.patient基础档案表（patient_id主键，存储注册填写健康信息、过敏史）；
2.department科室表、doctor医生表、schedule排班表、order挂号订单表；
3.chat医患对话记录表（绑定session_id、patient_id）；
4.clinic_record门诊病历表（存储医生录入四诊、辨证、处方）；
5.user用户表（患者/医生账号区分权限）、collection收藏表。
7.2API接口分层
1.用户层：注册、登录、病历建档、档案修改；
2.挂号层：科室排班查询、创建挂号订单、扫码接诊接口；
3.对话层：医患消息收发、历史聊天记录查询；
4.Agent调用层：接收前端`session_id/patient_id/user_input/mode/scene`，导入Agent函数获取答复；
5.图谱查询层：封装所有Cypher语句，提供药材/证型/医案/科室统一查询接口，供Agent、前端调用；
6.病历层：病历新增、查询。
7.3对接规范
1.接收前端结构化弹窗请求，透传给Agent并返回弹窗详情；
2.Nginx反向代理、全局跨域、请求日志、接口异常捕获。

十一、项目新增核心业务流程（可直接放进文档流程图说明）
1.患者流程：微信注册建档→AI预问诊采集症状→推荐科室/线上挂号→线下就诊→医患线上对话（包括开方、用量和后续身体症状沟通）；
2.医生流程：网页登录→接诊队列扫码调取患者病历→医患对话录入四诊信息→AI逐步提示缺失症状→信息充足自动诊断（一次图谱查询获取证型+方剂+药材+医案+科室）+自动推送相似药材/医案按钮→点击弹窗查看详情/手动自定义查询→确认方剂保存至患者电子病历。
